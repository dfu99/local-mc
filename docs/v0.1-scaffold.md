## Module-by-module honest assessment

### `lmc/config.py` — likely fine

Pure data + filesystem-path code. No external API surface. Should work
out of the box. The `LMC_HOME` env-var override is the test seam for
isolating runs.

### `lmc/projects.py` — likely fine

Read/write a YAML file. `Project.exists_on_disk()` is a one-liner. The
only failure mode is users editing the YAML by hand and producing
something `yaml.safe_load` can't parse — that surfaces as an exception,
which is acceptable.

### `lmc/store.py` — likely fine, **but unverified**

SQLite schema is small. The `connect()` context manager commits on
success, closes on exit. The streaming-friendly helpers
(`add_message` then `append_to_message`) are the part most likely to
have subtle bugs — verify with a test that streams 100 deltas into one
message and re-reads it.

### `lmc/sessions.py` — **risky, verify first**

This is the one to watch. Two concerns:

1. **The flag combination is unverified.** It uses `claude -p
   --output-format stream-json --input-format stream-json --verbose
   --resume <sid>`. Some of those flags may have been renamed or
   removed in the installed Claude Code version. Run
   `claude --help | grep -E '(input-format|output-format|resume)'`
   on the target machine before assuming it works.

2. **The stream-json event parser is best-effort.** It handles `system`
   (init), `assistant` (text + tool_use), `user` (tool_result), and
   `result` (done). Other event types are silently skipped, which is
   the right default but may mask issues. Add logging if turns go
   black-hole-quiet.

`EchoAgent` is the easy escape hatch: set `agent: echo` in
`settings.yaml` to bypass `claude` entirely.

### `lmc/artifacts.py` — likely fine

Pure stdlib. Snapshot / diff with `mtime` + glob. The default globs are
opinionated (`figures/`, `results/`, `*.png`, `*.pdf`, etc.) — adjust per
project via `settings.artifact_globs` if needed.

### `lmc/server.py` — mostly fine

Endpoints are straightforward. The piece that might bite:

- The `/api/files` sandbox uses `Path.resolve()` + `relative_to`. Symlinks
  inside a project that point outside won't be served, which is the
  desired behavior. Verify this is actually what `relative_to` does after
  `resolve` (it does, but write a test).
- The WebSocket loop is a single-receive-then-respond model. That's fine
  for one-at-a-time turns but means there's no way to cancel a running
  turn. Add a `cancel` event in v0.2.

### `lmc/cli.py` — likely fine

Plain argparse. The `serve` subcommand imports `uvicorn` lazily and
prints a helpful error if it's missing.

### `web/` — likely fine; visually unverified

Three files, no deps. Hand-rolled markdown renderer is small and only
covers code blocks, inline code, bold/italic, and links. It does *not*
support tables, lists, or headers — by design, since the agent's output
in this UI is conversational, not blog-style. If that's wrong, swap in
a tiny vendored renderer (e.g. snarkdown is ~500 bytes minified).

The websocket reconnect logic is **missing**. If you switch projects or
the server restarts, the client doesn't auto-reconnect; you need to
re-select the project. Add reconnect in v0.2.

### `bin/lmc` — likely fine

Runs `lmc.cli:main` after putting the repo root on `sys.path`. Useful
for `./bin/lmc serve` without installing the package.

## Things explicitly *not* in v0.1

- Anything mc-status-like (no batch view, no per-project last-activity
  panel).
- Anything AFK / queue / autochain.
- Reports (morning report, changelog, head scientist).
- Send-from-CLI (`lmc send <proj> "msg"` doesn't exist yet — the chat
  input in the browser is the only way to send a message).
- Tests + CI.
- PyPI publishing.

These are scoped in `../tasks/planning.md`.
